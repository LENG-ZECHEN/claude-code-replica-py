## Overview

This subsystem brings the replica's memory module up to the source-code
form: LLM-based sideQuery recall on the read path and an automatic,
opt-in post-turn extraction subloop on the write path, layered on top of
the `.md` + YAML-frontmatter store from `memory.py`. On each user turn the
agent surfaces up to 5 relevant memories — selected by a cheap LLM call,
with a lexical Jaccard fallback — as `<system-reminder>` ATTACHMENT
messages; after a turn that did not itself write memory, an extraction
runner reviews the new messages and persists anything worth keeping. It
deliberately does NOT thread the selector call asynchronously (the M-ε
async sideQuery is a deferred follow-up) and keeps the existing lexical
`MemorySelector` only as a fallback rather than as the primary read path.

## Boundaries

- **Reads from**: the `.md` memory store via `scan_memory_files`; the live `Transcript`; the user's latest input (as the recall query); `Provider.call_selector`.
- **Writes to**: the `.md` memory store (a fresh local `ProjectMemory(memory_dir)` during extraction; `write_memory_entry` during the main loop); the transcript (injected `ATTACHMENT_MEMORY` messages); the `memory_select` trace channel; `MetricsCollector`.
- **Depends on**: `memory.py`, `provider.py`, `models.py`, `coding_tools.py`, `tools.py`, `trace.py`, `transcript.py`. No new third-party dependency (frontmatter parsing stays hand-rolled; no PyYAML).
- **Does NOT do**: async/threaded sideQuery (M-ε, deferred); embedding or BM25 selection (lexical Jaccard fallback only); memory deletion from conversation (`/forget` out of scope — delete stays CLI-only via `simple-agent memory delete`).

## Known caveats (as shipped)

The end-to-end turn path has several wired-but-inert spots flagged in the
review (green unit tests exercise the pure functions directly, so these
did not fail CI):

- `recent_tools` passed to `find_relevant_memories` is always `[]` in the
  live loop — `inject_memory_attachments` runs after the new user message
  is appended, so `collect_recent_successful_tools` stops immediately.
- `read_file_state` (the already-Read dedup set) is never populated.
- The `memory_select` trace emits `fallback_used=False` unconditionally
  and reports `manifest_size == selected_count`.
- The `extraction_in_progress` re-entrancy gate (gate 4) is inert; only
  gate 1 (`is_subloop`) protects re-entrancy.
