**New dataclass `ExtractionResult`:**
```python
@dataclass(frozen=True)
class ExtractionResult:
    written_paths: tuple[str, ...]  # absolute paths of .md files written
    errors: tuple[str, ...]         # e.g. "max turns reached", tool errors
    turn_count: int
```

**`ExtractMemoriesRunner.__init__` signature:**
```python
class ExtractMemoriesRunner:
    def __init__(
        self,
        provider: Provider,
        memory_dir: Path,
        system_prompt: str,
        base_messages: list[Message],  # snapshot — do NOT mutate
        tool_registry: ToolRegistry,
    ) -> None:
```

**`ExtractMemoriesRunner.run(new_message_count: int) -> ExtractionResult`:**
1. Build the extraction prompt via `build_extract_prompt(new_message_count,
   format_memory_manifest(scan_memory_files(memory_dir)))`.
   (Note: `scan_memory_files` and `format_memory_manifest` will be added in M6;
   for now, use a stub that calls `ProjectMemory._get_manifest_content()` or
   just reads `MEMORY.md` if it exists, else empty string. The stub is fine for
   M4 — M6 will replace it when `memdir.py` exists.)
2. Build an inner message list: the extraction prompt as a user HumanMessage.
3. Loop up to MAX_TURNS=5:
   a. Call `provider.call(system=system_prompt, messages=messages, tools=whitelisted_tools)`.
   b. If stop_reason is NOT `tool_use`, break (extraction done).
   c. For each tool_use in the response:
      - If tool name NOT in whitelist: append tool_result with is_error=True,
        message `"Tool '{name}' is not available in the extraction context"`.
      - If tool name in whitelist: execute via `tool_registry.execute(tool_use)`,
        track `write_memory_entry` results.
   d. Append assistant + tool_result messages to inner list.
4. If turn count reaches MAX_TURNS and stop_reason is still `tool_use`:
   record `errors=["max turns reached"]`, stop without raising.
5. Return `ExtractionResult(written_paths=..., errors=..., turn_count=...)`.

**Tool whitelist** = `{"read_file", "list_files", "search_text", "write_memory_entry"}`.
Enforced by inspecting `tool_use.name` before calling `tool_registry.execute()`.
For `write_memory_entry`, also verify the resolved `file_path` is under `memory_dir`
(traversal defense, reusing `Path.is_relative_to` pattern from M1).

**`build_extract_prompt(new_message_count, existing_memories_manifest) -> str`:**
5-section extraction prompt:
```
You are a memory extraction agent. The user's agent just finished a conversation
turn ({new_message_count} new messages). Your job is to review those messages and
decide whether anything is worth saving to long-term memory.

**Immediate action**: Review the recent messages and extract any information
that should be saved using the write_memory_entry tool.

**Memory types to consider**:
- `user`: information about the user (role, preferences, expertise)
- `feedback`: guidance given about how to behave
- `project`: ongoing work context, goals, decisions, deadlines
- `reference`: pointers to external resources, docs, tickets

**Do NOT save**: code snippets, git history, ephemeral task details,
anything already in CLAUDE.md, speculative ideas not confirmed by the user.

**How to save**: use write_memory_entry(type, id, name, description, body).
If nothing is worth saving, respond with a plain text message explaining why.

**Existing memories** (do not duplicate):
{existing_memories_manifest}
```

This function lives in `extract_memories.py` as a module-level function,
not a method — it has no state dependency.

**M4 stub for manifest**: since `memdir.py` (M6) doesn't exist yet, use:
```python
def _get_existing_manifest(memory_dir: Path) -> str:
    manifest = memory_dir / "MEMORY.md"
    if manifest.exists():
        return manifest.read_text()[:2000]  # cheap truncation for now
    return "(no memories yet)"
```
M6 will provide `scan_memory_files` + `format_memory_manifest`; M7 will
wire them in as the canonical call. M4's stub is intentionally minimal.
