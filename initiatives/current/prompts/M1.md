Two things to fix/add while generalizing:

  1. **CONTEXT INJECTION GAP (real bug — fix it)**: `extract_memories.py`
     stores `self._base_messages = list(base_messages)` at line 118, but
     `run()` (line 122) builds `messages` from ONLY the task prompt
     (`messages = [{"role": "user", "content": prompt}]` at line 126) —
     `self._base_messages` is **never sent** to the provider. The
     conversation the runner was handed is silently dropped.
     `ForkedAgentRunner.run(task_prompt, context_messages=())` MUST
     prepend `context_messages` so they reach the sub-agent's FIRST
     provider call. Assert this directly:
     `MockProvider.history[0].messages` contains the injected
     context_messages. (Dream — M6 — passes none; it reads from disk via
     tools. SM-extraction — M3 — passes the post-last-summarized message
     slice. Both rely on this being fixed.)
  2. **PER-CALL GATE replaces the name-whitelist**: lift the current
     `_TOOL_WHITELIST` membership check (`extract_memories.py:41-46` +
     the `if tc.name not in _TOOL_WHITELIST` branch in `_execute_tool`,
     lines 184-188) into a `can_use_tool(name, input) -> (allow, reason)`
     callback so callers can express **path-scoped** rules (read-only ops
     + writes within a dir), not just a tool-name set. This is the SAME
     philosophy as plan-mode soft-deny in `loop.py::_execute_one`
     (`loop.py:900-921`): when the gate denies, return `is_error=True`
     with the gate's reason BEFORE the tool reaches `ToolExecutor` — the
     denied call must NEVER touch the executor/registry. Note the
     existing soft-deny does NOT filter the API `tools` schema (it stays
     byte-identical); mirror that — the gate is a runtime allow/deny, not
     a schema filter.
