### Implementation notes from PLAN

Source mapping (read BEFORE implementing — every claim below ties to a
specific TS line range in `claude-code-source-code/`):

  - `src/services/compact/sessionMemoryCompact.ts:58-60`
    DEFAULT_SM_COMPACT_CONFIG (minTokens=10_000, minTextBlockMessages=5,
    maxTokens=40_000) — the SM-compact eligibility config.
  - `src/services/SessionMemory/prompts.ts:11-41`
    DEFAULT_SESSION_MEMORY_TEMPLATE — the fixed markdown sections:
    Session Title / Current State / Task specification / Files and
    Functions / Workflow / Errors & Corrections / Documentation /
    Learnings / Key results / Worklog. The header + italic instruction
    lines are load-bearing (they are the template the LLM updater in M3
    will fill); the deterministic fold here populates the same fixed
    section set.
  - `src/services/SessionMemory/prompts.ts:256`
    truncateSessionMemoryForCompact — per-section cap
    MAX_SECTION_LENGTH=2000 tok; total 12_000. Apply an analogous
    per-section + total cap in the deterministic fold so a warm state
    cannot grow unbounded across turns.

Design intent (from PLAN):

  - The 9-section accumulator MIRRORS `RuleBasedSummarizer`'s existing
    section extraction (compact.py:124-191). REUSE those heuristics for
    the deterministic fold so a warm `SessionMemoryState` rendered to text
    is structurally identical to what `RuleBasedSummarizer.summarize()`
    would produce on the same messages. DRY: factor the shared section
    logic rather than copy-pasting it if that keeps both files ≤800 lines;
    if extraction cannot be cleanly shared without churn outside the
    Expected-files list, call the public
    `RuleBasedSummarizer().summarize()` from inside the fold and merge its
    output into the accumulator — do NOT reimplement the heuristics from
    scratch.
  - DEFAULT engine is LLM (M3 wires the LLM updater); the deterministic
    fold here is BOTH the no-API/test fallback AND the thing
    `SessionMemorySummarizer` reads at compaction time. M2 ships ONLY the
    deterministic fold + the summarizer that reads it — the LLM updater is
    M3.
  - `SessionMemorySummarizer(state, fallback)` is a producer/consumer
    split: `update_session_memory` is the PRODUCER (called per-turn in M3),
    `SessionMemorySummarizer.summarize()` is the CONSUMER (called at
    compaction). On a WARM state it returns the prewarmed text with ZERO
    provider calls (this is the entire point — O(0) compaction); on a
    COLD/empty state it delegates to the injected fallback
    (`RuleBasedSummarizer` or `LLMSummarizer`).

Concrete shape guidance:

  - `SessionMemoryState` is a frozen dataclass. Hold the 9 (effectively
    "9-section" per the exit gate; the TS template names 10 headings —
    fold them into the section set you commit to and document the exact
    set in the module docstring) sections as an immutable mapping
    (e.g. `sections: tuple[tuple[str, str], ...]` or a frozen field per
    section — pick one, justify in the docstring, keep it round-trippable).
    Add a `is_warm` / `is_empty` boolean property so the summarizer can
    branch cheaply.
  - `to_jsonable()` returns a plain `dict[str, Any]` (JSON-serializable);
    `from_jsonable(data)` reconstructs a `SessionMemoryState`, IGNORING
    unknown keys (forward-compat — a future M3/M4 field must not crash an
    old reader). Mirror the forward-compat posture already used by
    `transcript.from_jsonable` / `session_store` (absent/unknown keys are
    tolerated, never fatal).
  - `update_session_memory(state, new_messages)` MUST return a NEW
    `SessionMemoryState`; the input `state` and `new_messages` are never
    mutated. Apply the per-section + total cap from `prompts.ts:256` so a
    long session does not produce an unbounded warm summary.
  - `SessionMemorySummarizer.summarize(messages)` MUST match the
    `Summarizer` Protocol signature exactly (`summarize(self, messages:
    list[Message]) -> str`, compact.py:102-107) so it is a drop-in for
    `ContextCompactor(summarizer=...)`.
